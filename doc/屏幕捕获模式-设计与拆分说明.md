# 屏幕捕获模式设计与实现说明

## 1. 文档目的

本文档用于说明“屏幕捕获模式”从 UI 临时实现到后端化重构的设计过程、当前落地状态、接口边界与已知限制，作为后续维护与扩展（如区域捕获、窗口捕获、录制）的开发依据。

---

## 2. 背景与重构动机

### 2.1 初始问题

在最初版本中，屏幕捕获逻辑直接放在 `CaptureWindow` 内部，这会导致：

- `widget/src/capturewindow.cpp` 责任过重（模式切换 + 业务调度 + 采集实现）
- UI 层与采集实现耦合，难以复用
- 后续增加 FFmpeg 方案时，无法形成统一抽象

### 2.2 重构目标

将屏幕采集能力下沉到媒体后端体系，与相机/播放器一致：

- API 层定义能力抽象
- Runtime 负责按后端类型构造实现
- Service 负责业务层封装
- Widget 仅负责交互和展示

---

## 3. 分层设计（最终版）

### 3.1 API 层

- `api/include/fplayer/api/media/iscreencapture.h`
- `api/src/iscreencapture.cpp`

定义：

- `ScreenDescription`：屏幕元数据（名称、主副屏、像素分辨率、坐标）
- `IScreenCapture`：统一接口（枚举/选择/绑定预览/启停/鼠标能力）

### 3.2 后端实现层

- Qt 实现  
  - `backend/media_qt6/include/fplayer/backend/media_qt6/screencaptureqt6.h`
  - `backend/media_qt6/src/screencaptureqt6.cpp`

- FFmpeg 实现（gdigrab）  
  - `backend/media_ffmpeg/include/fplayer/backend/media_ffmpeg/screencaptureffmpeg.h`
  - `backend/media_ffmpeg/src/screencaptureffmpeg.cpp`

### 3.3 Runtime 层

- `runtime/include/fplayer/runtime/runtime.h`
- `runtime/src/runtime.cpp`

新增能力：

- `createScreenCapture(MediaBackendType backend)`
- `bindScreenPreview(const PreviewTarget& target)`

### 3.4 Service 层

- `service/include/fplayer/service/service.h`
- `service/src/service.cpp`

新增能力：

- 初始化：`initScreenCapture(...)`
- 绑定预览：`bindScreenPreview(...)`
- 列表/选择：`getScreenList()`、`selectScreen(...)`
- 控制：`screenSetActive(...)`、`screenIsActive()`
- 鼠标能力：`screenSetCursorCaptureEnabled(...)`、`screenCanControlCursorCapture()`

### 3.5 Widget 层

- `widget/src/capturewindow.cpp`

职责回归：

- 模式切换
- 控件显隐与布局
- 将用户操作透传给 `Service`

删除了早期 UI 层临时控制器：

- `widget/include/fplayer/widget/screencapturecontroller.h`
- `widget/src/screencapturecontroller.cpp`

---

## 4. 为什么“Qt 版也要搬到后端”

是的，必须搬。原因如下：

- 若 FFmpeg 在后端、Qt 在 widget，会产生“双实现双入口”并导致维护混乱
- 相同业务能力（屏幕采集）应通过同一抽象（`IScreenCapture`）暴露
- 便于运行时切换后端，UI 不关心实现细节
- 后续测试、日志、故障定位统一落在 service/runtime/backend 链路

结论：Qt 与 FFmpeg 屏幕采集都应属于后端实现，widget 仅做调用。

---

## 5. 运行时流程（屏幕模式）

### 5.1 进入屏幕模式

1. `CaptureWindow` 切换模式为 `Screen`
2. 选择渲染后端（当前默认 FFmpeg）
3. `Service::bindScreenPreview()` 绑定预览目标
4. `Service::getScreenList()` 刷新下拉
5. `Service::selectScreen(index)` 选择屏幕
6. `Service::screenSetActive(true)` 开始采集

### 5.2 切换屏幕

1. 下拉变化触发 `selectScreen(index)`
2. 后端停旧采集并重建输入
3. 继续推流到当前预览窗口

### 5.3 播放/暂停按钮

- 屏幕模式下映射到 `screenSetActive(true/false)`

---

## 6. 鼠标指针捕获设计

### 6.1 Qt 后端

当前 Qt 头文件中未公开 `QScreenCapture::setCursorCaptureEnabled`。  
因此 Qt 实现通过能力探测决定是否支持：

- 支持：运行时调用
- 不支持：返回 false，由 UI 禁用勾选并提示

### 6.2 FFmpeg 后端

FFmpeg `gdigrab` 通过参数控制：

- `draw_mouse=1`：采集鼠标指针
- `draw_mouse=0`：不采集

当前在 `ScreenCaptureFFmpeg` 中已接入该参数，并在切换开关时重建采集输入使其生效。

---

## 7. 分辨率与坐标规则

为避免高 DPI 下“逻辑分辨率/像素分辨率”混用，统一采用像素坐标：

- `width = logicalWidth * devicePixelRatio`
- `height = logicalHeight * devicePixelRatio`
- `x/y` 同样按 `devicePixelRatio` 换算

用途：

- 下拉显示分辨率
- FFmpeg `gdigrab` 的 `video_size` / `offset_x` / `offset_y` 参数

---

## 8. 屏幕下拉宽度策略

屏幕模式下，宽度不足通过三层策略修正：

- 根据最长文本计算目标宽度
- 同步设置 `QComboBox` 本体宽度与内容长度策略
- 设置 popup 最小宽度，避免下拉项被截断

并在屏幕模式压缩左侧 spacer，优先保证屏幕下拉可读性。

---

## 8.1 屏幕帧率策略（动态）

为避免“固定 60 FPS 上限”与高刷屏不匹配，屏幕模式改为动态帧率列表：

- 根据当前屏幕刷新率生成可选档位（如 75/90/120/144/165/240）
- 结合屏幕物理像素分辨率（含 DPR）计算推荐帧率
- 下拉项会标记推荐值：`xx FPS (推荐)`
- 每块屏幕单独记忆用户最后一次选择，切回该屏幕时优先恢复历史值

实现要点：

- UI 层新增 `refreshScreenFpsUi(screenIndex)`，按屏幕重建候选帧率
- `preferredFpsForScreen(screenIndex)` 综合分辨率与刷新率上限给出推荐
- `selectScreen(index)` 中先更新帧率列表，再设置后端采集帧率并开始采集

后端能力：

- FFmpeg 后端支持动态帧率（`gdigrab framerate`）
- Qt 后端当前仍标记为“不支持帧率控制”，UI 会禁用并提示

---

## 9. 当前默认配置

- 屏幕采集默认后端：`FFmpeg`
- 原因：鼠标捕获可控（`draw_mouse`），行为更可预期

如需切回 Qt 屏幕后端，可在 `CaptureWindow` 初始化时将 `m_screenBackendType` 改为 `MediaBackendType::Qt6`。

---

## 10. 已知限制

- Qt 后端在当前环境下可能无法控制鼠标捕获（取决于 Qt API 暴露）
- FFmpeg `gdigrab` 在高负载场景可能带来更高 CPU 占用（后续可加帧率与缩放选项）
- 目前尚未支持“窗口捕获/区域捕获”，仅支持按屏幕捕获

---

## 11. 后续建议

- 增加“屏幕后端选择”设置项（Qt / FFmpeg）
- 增加采集参数配置（帧率、是否缩放、目标分辨率）
- 增加窗口捕获与区域捕获
- 增加屏幕采集错误码与 UI 反馈（权限不足、设备不可用、初始化失败）

## 12. 本次开发问题记录（编译/链接）

本节记录本轮改造过程中出现的真实问题，便于后续快速定位同类故障。

### 12.1 `QScreenCapture::setCursorCaptureEnabled` 编译失败

- **现象**  
  `error C2039: "setCursorCaptureEnabled": 不是 "QScreenCapture" 的成员`
- **根因**  
  当前使用的 Qt 头文件未暴露该 API（不同 Qt 版本/平台能力差异）。
- **处理**  
  改为运行时能力探测（反射调用）：  
  - 若方法存在则调用  
  - 若不存在则返回 `false`，由 UI 禁用勾选并提示
- **结论**  
  不再依赖该 API 的编译期存在性，兼容性更好。

### 12.2 `QObject::connect` 参数重载匹配失败

- **现象**  
  `error C2665: QObject::connect: 没有重载函数可以转换所有参数类型`
- **根因**  
  连接类型写成按位组合后被推导为 `int`，与期望的 `Qt::ConnectionType` 重载不匹配。
- **处理**  
  改为安全写法：先 `disconnect`，再使用 `Qt::QueuedConnection` 重连。
- **结论**  
  解决重载匹配问题，同时避免重复连接。

### 12.3 `ScreenCaptureQt6` 的 moc 符号链接失败

- **现象**  
  `LNK2001: metaObject / qt_metacast / qt_metacall / staticMetaObject`
- **根因**  
  类启用了 `Q_OBJECT`，但该类并未实际需要元对象功能，且构建链路中出现 moc 符号不匹配。
- **处理**  
  移除 `Q_OBJECT`（该类无自定义 signals/slots 需求）。
- **结论**  
  消除对 moc 生成符号的依赖，链接恢复正常。

### 12.4 `ScreenCaptureFFmpeg` 的 moc + signal 链接失败

- **现象**  
  `LNK2001/LNK2019: ScreenCaptureFFmpeg::metaObject / yuvFrameReady / staticMetaObject`
- **根因**  
  `Q_OBJECT + signals` 依赖 moc 生成；在当前构建链路下未形成稳定链接。
- **处理**  
  去除 `Q_OBJECT` 与 `signals`，改为：
  - 采集线程中生成帧后
  - 通过 `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` 投递到 `FGLWidget::updateYUVFrame`
- **结论**  
  不依赖 moc 也可实现线程安全的 UI 线程更新，链接问题彻底规避。

### 12.5 `runtime` 编译阶段找不到 `QMediaCaptureSession`

- **现象**  
  `fatal error C1083: 无法打开包括文件: “QMediaCaptureSession”`
- **根因**  
  `runtime.cpp` 间接包含 `screencaptureqt6.h`，该头直接包含 QtMultimedia 头，导致 `runtime` 目标也被迫依赖 QtMultimedia 头路径。
- **处理**  
  在 `screencaptureqt6.h` 中改为前置声明：  
  - `class QMediaCaptureSession;`
  - `class QScreenCapture;`  
  并将真实头包含移到 `screencaptureqt6.cpp`。
- **结论**  
  降低头文件耦合，避免跨模块传染式依赖。

### 12.6 经验与规范沉淀

- 后端头文件尽量避免直接包含重量级 Qt 模块头，优先前置声明。
- 若类不需要元对象能力，不使用 `Q_OBJECT`。
- 在当前工程中，新增 QObject 派生类时应优先使用“无 signals + invokeMethod”方案，减少 moc 链路不稳定带来的风险。
- 跨层模块（runtime/service）尽量只依赖抽象接口头，不依赖具体后端实现细节头。

### 12.7 屏幕模式底部工具栏对齐异常

- **现象**  
  屏幕捕获模式下底部工具栏视觉上接近“居中”，与摄像头/文件模式不一致。
- **根因**  
  为给屏幕下拉让位，在模式切换逻辑中对 `horizontalSpacer` 动态改成了固定宽度，破坏了原有“伸缩 spacer 推右对齐”的布局策略。
- **处理**  
  删除屏幕模式/摄像头模式切换时对 `horizontalSpacer->changeSize(...)` 的动态修改，恢复统一布局规则。
- **结论**  
  三种模式底部工具栏对齐方式重新一致；屏幕下拉宽度自适应仍保留。
