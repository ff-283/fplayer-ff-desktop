# 推拉流使用说明

## 1. 功能入口

在主窗口顶部菜单栏中，点击：

- `推拉流` -> `推流`
- `推拉流` -> `拉流`

会弹出对应的配置窗口。

---

## 2. 推流使用

## 2.1 配置项说明

- `协议模板`：快速填充输出地址（RTMP / RTSP / SRT）
- `输出`：推送目标地址
- `帧率`：`跟随当前` 或手动指定
- `尺寸`：`跟随当前`、预设分辨率、或手动输入 `WxH`
- `保持宽高比`：按源分辨率比例自动调整目标尺寸
- `码率`：`跟随当前` 或手动指定（kbps）
- `视频编码器`：`自动（优先NVENC/AMF）` / `CPU（x264）` / `NVIDIA NVENC` / `AMD AMF`
- `声音来源`：可选 `关闭声音` / `系统声音（实验）` / 枚举到的本机音频输入设备

常见示例：

- RTMP：`rtmp://127.0.0.1:1935/live/stream`
- RTSP：`rtsp://127.0.0.1:8554/live/stream`
- SRT：`srt://127.0.0.1:8890?mode=caller`

## 2.2 操作流程

1. 点击 `推拉流` -> `推流`
2. 根据当前模式确认来源（文件/摄像头/屏幕）
3. 选择协议模板（可选）
4. 填写输出地址与推流参数
5. 点击 `开始推流`（窗口不会关闭）
6. 如需停止，点击窗口内 `停止推流`

## 2.3 状态与日志

窗口内提供：

- `状态`：运行中 / 已停止 + 退出码（与程序内 FFmpeg 库返回的错误码一致，成功为 `0`）
- `日志`：应用侧记录的最近状态与错误摘要（自动刷新）
  - 启动后会打印“最终生效参数”，用于核对实际输出（输入尺寸、输出尺寸、FPS、码率、模式）
  - 会打印当前视频编码器，例如：`编码器=h264_nvenc（硬件）`、`编码器=h264_amf（硬件）`、`编码器=libx264（软件）`
  - 屏幕模式会额外打印来源链路：
    - `来源=DXGI预览帧`：屏幕视频来自当前 DXGI 预览帧（推荐）
    - `采集区域=...`：来自 FFmpeg `gdigrab` 采集链路

---

## 3. 拉流使用

## 3.1 配置项说明

- `输入`：源流地址
- `协议模板`：快速填充输入地址（RTMP / RTSP / SRT）
- `输出`：输出文件路径（不勾选预览时生效）
- `直接预览到播放窗口`：勾选后不落地文件，直接在主播放窗口预览

常见示例：

- 输入流：`rtmp://127.0.0.1:1935/live/stream`
- 输出文件：`D:/temp/pull.mp4`

## 3.2 操作流程（落地文件）

1. 点击 `推拉流` -> `拉流`
2. 填写输入流地址
3. 填写输出文件（可用“选择输出文件”）
4. 不勾选“直接预览到播放窗口”
5. 点击 `确定` 开始拉流
6. 如需停止，点击 `停止拉流`

## 3.3 操作流程（直接预览）

1. 点击 `推拉流` -> `拉流`
2. 填写输入流地址
3. 勾选 `直接预览到播放窗口`
4. 点击 `确定`
5. 主窗口切换到播放器预览逻辑，直接显示流内容

---

## 4. 历史记录

推流和拉流都支持最近地址记录（会话内）：

- 最近推流输入/输出
- 最近拉流输入/输出

每项最多保留 8 条，最新项在最前面。

## 5. 常见问题

## 5.0 推流前置条件

- 若提示 `stream not initialized`，通常是构建时未启用网络 FFmpeg 后端
- 请确认 CMake 选项：`FPLAYER_BUILD_NET_FFMPEG=ON`
- 重新 Configure + Build 后再运行

## 5.1 点击确定后提示启动失败

可能原因：

- 程序目录下 **缺少 FFmpeg 运行库**（`avformat-*.dll`、`avcodec-*.dll` 等），或安装不完整
- 输入地址不可达、协议/格式不被当前编译的 FFmpeg 支持
- 输出地址不可写、媒体服务器拒绝连接或路径错误

建议：

- 确认与主程序一同发布的 `3rd/ffmpeg_*` 对应 **bin** 中 DLL 已复制到可执行文件目录（与正常播放、采集功能一致）
- 检查地址、端口、防火墙与服务器是否已启动

## 5.2 推流成功但播放器看不到画面

建议排查：

- 推流协议与播放端协议是否一致
- 媒体服务器是否正常（如 rtmp/rtsp 服务）
- 目标 URL 路径是否正确

## 5.3 拉流预览无画面

建议排查：

- 输入流是否可用（先用 ffplay/vlc 验证）
- 网络是否连通
- 主窗口当前是否被其他模式占用（重复切换模式后重试）

## 5.4 屏幕推流看起来忽快忽慢

建议排查顺序：

1. 先用 `跟随当前` 的 FPS/码率
2. 若机器负载高，降低 FPS（如 60 -> 30）
3. 码率过低会“糊”，可提高到合适区间（例如 1080p60 常见 8~16Mbps）
4. 对比日志里的“采集尺寸/输出尺寸/FPS/码率”是否符合预期

## 5.5 想确认是缩放还是裁剪

查看推流日志中的两组值：

- `采集区域=... WxH`
- `输出=... WxH`

当采集尺寸与输出尺寸不同且画面完整时，表示为“整屏采集后缩放输出”。

## 5.6 屏幕推流声音相关说明

- 当前版本已提供“声音来源”下拉（包含“关闭声音”）
- 当选择“关闭声音”时，视频推流正常按无音轨处理
- 屏幕推流走 `FFmpeg gdigrab` 链路时，音频会做 `解码 -> 重采样 -> AAC 编码` 后复用，避免 FLV/RTMP 下 `-22(EINVAL)` 的常见兼容问题
- 屏幕推流走 `DXGI预览帧` 链路时，当前仅推视频，音频参数会被忽略并在日志中提示
- 若你必须同时推系统声音，可临时切换到非 DXGI 屏幕后端后再推流

## 5.7 硬件编码器说明（NVENC / AMF）

- 选择 `自动` 时，系统按 `NVENC -> AMF -> CPU` 顺序自动回退
- 若手动选择 `NVIDIA NVENC` 或 `AMD AMF`，但本机 FFmpeg 未编译该编码器或驱动不可用，会启动失败并提示编码器不可用
- 建议通过日志中的 `编码器=...（硬件/软件）` 确认是否真正走了硬件编码
