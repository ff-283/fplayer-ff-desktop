# 推拉流使用说明

## 1. 功能入口

在主窗口顶部菜单栏中，点击：

- `推拉流` -> `推流`
- `推拉流` -> `拉流`

会弹出对应的配置窗口。

---

## 2. 推流使用

## 2.1 配置项说明

- `协议模板`：快速填充输出地址（RTMP / RTSP / SRT）
- `输出`：推送目标地址
- `帧率`：`跟随当前` 或手动指定
- `尺寸`：`跟随当前`、预设分辨率、或手动输入 `WxH`
- `保持宽高比`：按源分辨率比例自动调整目标尺寸
- `码率`：`跟随当前` 或手动指定（kbps）
- `视频编码器`：`自动（优先NVENC/AMF）` / `CPU（x264）` / `NVIDIA NVENC` / `AMD AMF`
- `声音来源`：可选 `关闭声音` / `系统声音（实验）` / 枚举到的本机音频输入设备

常见示例：

- RTMP：`rtmp://127.0.0.1:1935/live/stream`
- RTSP：`rtsp://127.0.0.1:8554/live/stream`
- SRT：`srt://127.0.0.1:8890?mode=caller`

## 2.2 操作流程

1. 点击 `推拉流` -> `推流`
2. 根据当前模式确认来源（文件/摄像头/屏幕）
3. 选择协议模板（可选）
4. 填写输出地址与推流参数
5. 点击 `开始推流`（窗口不会关闭）
6. 如需停止，点击窗口内 `停止推流`

## 2.3 状态与日志

窗口内提供：

- `状态`：运行中 / 已停止 + 退出码（与程序内 FFmpeg 库返回的错误码一致，成功为 `0`）
- `日志`：应用侧记录的最近状态与错误摘要（自动刷新）
  - 启动后会打印“最终生效参数”，用于核对实际输出（输入尺寸、输出尺寸、FPS、码率、模式）
  - 会打印当前视频编码器，例如：`编码器=h264_nvenc（硬件）`、`编码器=h264_amf（硬件）`、`编码器=libx264（软件）`
  - 屏幕模式会额外打印来源链路：
    - `来源=DXGI预览帧`：屏幕视频来自当前 DXGI 预览帧（推荐）
    - `采集区域=...`：来自 FFmpeg `gdigrab` 采集链路
  - 屏幕模式“当前参数”会显示当前屏幕后端：`DXGI` / `FFmpeg(gdigrab)`

## 2.4 组合模式

- **入口**：在模式菜单选择「组合模式」（与摄像头/文件/屏幕并列）。展开后左侧可添加文件、摄像头、屏幕等来源，中间为画布预览（可拖动、叠放顺序、裁剪等），底部参数随当前选中来源切换。
- **输出内容**：推流画面为 **后台按场景合成后的视频**（多源 YUV 合成再编码），与预览布局、层级一致；**不是**对预览窗口截图推流。
- **分辨率与比例**：推流对话框中的「尺寸」若与左侧画布比例不一致，编码时会 **等比缩放画布并加黑边** 以适配输出分辨率，避免画面被横向或纵向拉伸；若希望铺满且无黑边，请选择与当前画布比例一致的输出尺寸（例如画布 16:9 时选 1920x1080）。
- **画布比例**：左侧可选择固定宽高比，主窗口可随比例调整；素材可独立是否锁定宽高比缩放。
- **播放/暂停**：**只有**主窗口底部工具栏一组播放/暂停图标；在组合模式下，该图标表示 **当前选中素材**（左侧列表或画布点击）自己的播放/暂停状态，切换选中行时图标会随之切换；不会一键暂停全部素材。
- **切换到其他模式**：会 **暂停/停止** 组合内所有素材的采集与播放，并收起组合画布（素材列表与子窗口保留，再次进入组合模式可继续编辑；不会一直在后台跑 DXGI/摄像头/文件解码）。
- **多路屏幕**：多路屏幕素材按各自播放状态独立运行，不再互相自动暂停；切换某一路的显示器会同步更新该素材名称与参数。
- **多路摄像头**：多路摄像头素材按 `sourceId` 独立发布帧，推流端不会再出现“两个摄像头素材互相闪现切换”。
- **素材命名**：左侧列表在设备切换后会更新名称，并附加短 `sourceId`（如 `[1a2b3c4d]`）便于区分同类素材。
- **音频**：组合场景推流已支持 `audio_in / audio_out` 音频链路（含回退与混音策略），与屏幕/摄像头推流保持同类解耦方案。
- **排查**：若需分析合成或编码问题，可让开发者在 `StreamFFmpeg::pushComposeSceneLoop`、DXGI/摄像头等位置 **临时打开** 已注释的诊断日志。

## 2.5 预览暂停后开始推流（单路：文件 / 摄像头 / 屏幕）

- **屏幕（DXGI 预览帧）与摄像头（预览帧推流）**：若在主界面先 **暂停预览** 再点推流，拉流端应能持续收到 **定格在暂停瞬间的画面**（推流侧会按帧率重复最后一帧），而不会长时间卡在加载中。
- **文件模式**：默认 **直推 copy** 或 **独立文件转码** 与预览里的播放器 **不是同一条管线**，推流内容与「预览是否暂停、播到哪一帧」**不一定一致**；若你期望推流画面与预览暂停时完全一致，需使用组合等能走预览帧总线的能力（详见开发文档说明）。

## 3. 拉流使用（监视窗口）

## 3.1 配置项说明

- `协议模板`：快速生成地址前缀（RTMP / RTSP / SRT）
- `推流码`：与协议模板拼接生成拉流地址
- `拉流地址`：只读标签（不可编辑），右侧可一键复制
- `输入`：实际拉流输入地址（默认可用自动生成地址，亦可手改）
- `输出`：输出文件路径（拉流任务输出）
- `开始拉流 / 结束拉流`：按钮互斥（开始前禁用结束；开始后禁用开始）

常见示例：

- 输入流：`rtmp://127.0.0.1:1935/live/stream`
- 输出文件：`D:/temp/pull.mp4`

## 3.2 操作流程

1. 点击 `推拉流` -> `拉流` 打开**非模态**监视窗口（不会阻塞主界面）
2. 选择协议模板并填写推流码，按需复制生成的拉流地址
3. 确认输入地址与输出文件
4. 点击 `开始拉流`
5. 成功后自动弹出独立 `拉流预览` 窗口
6. 如需停止，点击监视窗口 `结束拉流`（会联动关闭预览）

## 3.3 拉流预览窗口

- 预览窗口无菜单栏，底部仅保留 `暂停`、`刷新`、`音量` 滑块
- 关闭预览窗口 = 停止拉流任务
- 停止拉流任务 = 关闭预览窗口
- 与监视窗口双向联动

---

## 4. 历史记录

推流和拉流都支持最近地址记录（会话内）：

- 最近推流输入/输出
- 最近拉流输入/输出

每项最多保留 8 条，最新项在最前面。

## 5. 常见问题

## 5.1 屏幕模式出现扭曲/错位（尤其在 HDR 开启时）

可能表现：

- DXGI 预览时画面扭曲、闪烁或不稳定
- 日志提示 `skip unsupported duplication format=R16G16B16A16_FLOAT`

原因：

- 系统 HDR/高级色彩开启后，DXGI 可能返回浮点格式帧；当前策略会跳过这类不支持帧以保证进程稳定。

处理建议：

1. 在弹窗提示“已检测到系统HDR打开，为确保稳定，将屏幕获取后端改为ffmepg。是否切换？”时选择“是”
2. 或手动关闭系统 HDR 后继续使用 DXGI
3. 在推流窗口“当前参数”确认当前后端是否符合预期

## 5.0 推流前置条件

- 若提示 `stream not initialized`，通常是构建时未启用网络 FFmpeg 后端
- 请确认 CMake 选项：`FPLAYER_BUILD_STREAM_FFMPEG=ON`
- 重新 Configure + Build 后再运行

## 5.2 点击确定后提示启动失败

可能原因：

- 程序目录下 **缺少 FFmpeg 运行库**（`avformat-*.dll`、`avcodec-*.dll` 等），或安装不完整
- 输入地址不可达、协议/格式不被当前编译的 FFmpeg 支持
- 输出地址不可写、媒体服务器拒绝连接或路径错误

建议：

- 确认与主程序一同发布的 `3rd/ffmpeg_*` 对应 **bin** 中 DLL 已复制到可执行文件目录（与正常播放、采集功能一致）
- 检查地址、端口、防火墙与服务器是否已启动

## 5.3 推流成功但播放器看不到画面

建议排查：

- 推流协议与播放端协议是否一致
- 媒体服务器是否正常（如 rtmp/rtsp 服务）
- 目标 URL 路径是否正确

## 5.4 拉流预览无画面

建议排查：

- 输入流是否可用（先用 ffplay/vlc 验证）
- 网络是否连通
- 输入流地址是否有效（可先用 VLC/ffplay 验证）
- 监视窗口日志中是否有连接或解复用错误

## 5.5 屏幕推流看起来忽快忽慢

建议排查顺序：

1. 先用 `跟随当前` 的 FPS/码率
2. 若机器负载高，降低 FPS（如 60 -> 30）
3. 码率过低会“糊”，可提高到合适区间（例如 1080p60 常见 8~16Mbps）
4. 对比日志里的“采集尺寸/输出尺寸/FPS/码率”是否符合预期

## 5.6 想确认是缩放还是裁剪

查看推流日志中的两组值：

- `采集区域=... WxH`
- `输出=... WxH`

当采集尺寸与输出尺寸不同且画面完整时，表示为“整屏采集后缩放输出”。

## 5.7 推流音频说明（屏幕/摄像头/组合）

- 支持 `audio_in / audio_out` 单路或双路采集（失败自动降级）
- 非文件推流统一采用“视频线程 + 音频线程”解耦
- 音频发送采用固定节拍与补静音策略，降低“慢半拍/断续”风险
- 组合模式同样支持音频链路，不再默认无声

## 5.8 硬件编码器说明（NVENC / AMF）

- 选择 `自动` 时，系统按 `NVENC -> AMF -> CPU` 顺序自动回退
- 若手动选择 `NVIDIA NVENC` 或 `AMD AMF`，但本机 FFmpeg 未编译该编码器或驱动不可用，会启动失败并提示编码器不可用
- 建议通过日志中的 `编码器=...（硬件/软件）` 确认是否真正走了硬件编码
