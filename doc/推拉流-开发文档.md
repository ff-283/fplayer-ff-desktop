# 推拉流开发文档

## 1. 目标与范围（2026-04 更新）

本文档描述当前“推拉流”功能的工程落地方式，包括：

- 分层架构
- 关键接口
- UI 交互流程
- FFmpeg 库侧实现策略
- 已知限制与后续扩展建议

当前版本聚焦“可用闭环 + 可调优 + 分层可维护”，在进程内通过 **FFmpeg 动态库（libavformat / libavcodec / libavutil / libswscale / libswresample / libavdevice）** 实现推拉流，**不依赖**本机 `ffmpeg` 可执行文件或命令行。

## 2.2 后端实现层（FFmpeg）

文件：

- `backend/stream_ffmpeg/include/fplayer/backend/stream_ffmpeg/streamffmpeg.h`
- `backend/stream_ffmpeg/src/streamffmpeg.cpp`
- `backend/stream_ffmpeg/src/streamffmpeg_helpers.h/.cpp`
- `backend/stream_ffmpeg/src/audio_pipeline.h/.cpp`
- `backend/stream_ffmpeg/CMakeLists.txt`

实现类：`fplayer::StreamFFmpeg`

实现策略（当前）：

- 文件直推默认使用 **copy/remux**（低开销）
- 当配置了码率/尺寸/帧率时，文件模式切换为 **转码推流**（解码 -> 缩放 -> 编码 -> 推流）
- 屏幕模式支持两条视频链路：
  - `__screen_preview__`：消费 DXGI 预览帧总线（推荐，降低采集开销与鼠标闪烁）
  - `__screen_capture__`：FFmpeg `gdigrab` 采集链路（回退）
- 屏幕推流链路与当前屏幕采集后端联动：
  - 后端为 `Dxgi` 时优先走 `__screen_preview__`
  - 后端为 `FFmpeg` 时走 `__screen_capture__`
- 摄像头模式默认走“当前预览帧推流”，避免设备二次占用冲突
- 在 **独立工作线程** 中执行循环，避免阻塞 UI；`stop()` 通过 `AVFormatContext::interrupt_callback` 与原子标志请求中断
- 启动时调用 `avformat_network_init()`（仅一次）；输入侧在 `avformat_open_input` 前预分配 `AVFormatContext` 以便尽早挂上中断回调
- `recentLog` 为应用侧追加的简短状态/错误说明（带长度上限，避免无限增长）；`lastExitCode` 为 libav 错误码（成功为 `0`）

封装与编码策略：

- **推流**：输出封装为 FLV（RTMP/多数直播链路友好）
- **拉流**：由输出路径/协议自动推断容器
- **编码器**：优先 H.264，不可用时回退 MPEG4
- **视频编码器选择**：
  - UI 参数 `videoEncoder` 支持 `auto / cpu / nvenc / amf`
  - `auto` 策略：`h264_nvenc -> h264_amf -> libx264/H264/MPEG4`
  - 日志会输出实际命中编码器及硬件/软件属性
- **实时参数**：支持 `fps / size / bitrate`，并提供“跟随当前”策略
- **音频来源参数**：推流侧仅使用 `audio_in / audio_out`（输入/输出设备分别指定）
- **屏幕音频编码**：`gdigrab` 链路下音频走 `decode + swresample + AAC encode + mux`，避免 FLV 对输入音频格式不兼容导致 `-22`
- **非文件推流音视频解耦**：`__screen_preview__ / __screen_capture__ / __camera_preview__ / __camera_capture__` 均采用“视频线程 + 音频线程”并行采集/编码，复用统一时间基写入复用器
- **双设备音频混音**：非文件推流链路可同时采集输入设备（麦克风）与输出设备（系统回采），经双 FIFO 对齐后混成单路 AAC 推出
- **音频诊断日志**：屏幕推流每秒输出主/副路读帧数、样本数、FIFO 深度与采集模式（dshow/wasapi），用于快速定位“输出设备无声”
- **时间戳策略**：屏幕/转码路径优先使用源帧时间戳并做单调修正，减轻播放端“忽快忽慢”
- **结构拆解（2026-04）**：
  - `streamffmpeg_helpers.*` 承载参数解析、推流场景路由、编码器候选选择与通用诊断逻辑
  - `audio_pipeline.*` 承载音频平面混音、第二路 FIFO 混音，以及“编码帧填充（含尾包补静音）”可复用处理，降低多场景重复代码
  - `StreamFFmpeg::startPush/startPull` 仅保留编排，线程启动细节下沉为私有函数

说明：文件模式默认不重编码；其它模式按配置与场景走编码链路。

联调配置示例（非文件推流）：

- 仅麦克风输入：`audio_in=麦克风阵列 (Realtek...)`，`audio_out=` 置空
- 仅系统声音：`audio_out=扬声器 (Realtek...)`，`audio_in=` 置空
- 双设备混音：`audio_in=麦克风阵列 (Realtek...)` + `audio_out=扬声器 (Realtek...)`

降级策略：

- 双路模式下单路失败 -> 自动降级单路继续推流
- 双路都失败 -> 自动降级无声推流
- 文件模式不使用 `audio_in/audio_out`，沿用源内建音视频

## 3. 关键流程

## 3.1 推流流程

1. 用户打开 `推流` 对话框
2. 配置输出地址与推流参数（fps/尺寸/码率/视频编码器/宽高比/输入设备/输出设备）
3. 调用 `Service::streamStartPushByScene(...)`
4. Service 根据场景组装推流请求并转发到 `IStream::startPush(...)`
5. `StreamFFmpeg` 在后台线程按策略执行（copy / transcode / capture / preview-frame）
6. UI 定时拉取状态与日志展示（包含“最终生效参数”）

## 3.2 拉流流程（文件落地）

1. 用户打开 `拉流` 对话框
2. 配置输入流与输出文件
3. 调用 `Service::streamStartPull(...)`
4. `StreamFFmpeg` 在后台线程中拉流并写入目标文件（封装由输出路径推断）

## 3.3 拉流流程（直接预览）

1. 用户勾选 `直接预览到播放窗口`
2. 不走 `streamStartPull` 落盘流程
3. 直接调用现有播放器链路 `openMediaFile(inputUrl)` 进入播放预览

说明：该预览流程与“屏幕捕获模式”共用同一预览控件体系；当前 Windows 下屏幕采集默认后端为 `DXGI`。当 Service 检测到屏幕后端为 `Dxgi` 时，屏幕推流会自动切换到 `DXGI预览帧 -> FFmpeg编码推流` 链路。若用户因 HDR 选择回退到 FFmpeg 屏幕后端，则推流自动切换为 `gdigrab` 链路。

## 5. 设计取舍

## 5.1 为何在进程内使用 libav* 而非调用 ffmpeg 进程

- 与项目已集成的 FFmpeg 动态库 **统一依赖**，安装包无需再携带或查找 `ffmpeg.exe`
- 行为在应用内可控，便于与 **中断、错误码、后续扩展（编码参数、多任务）** 结合
- 仍可通过 `lastError` / `lastExitCode` 对照 libav 错误排查问题

## 5.2 为何拉流预览复用播放器链路

- 避免重复实现网络解复用与渲染路径
- 复用现有 `player` 预览能力，集成成本最低

## 6. 已知限制

- 推流输出侧当前固定以 FLV 创建输出上下文；若需 RTSP/SRT 专用封装策略仍需继续细化
- 文件转码推流当前为“视频转码 + 音频流 copy（若源存在音频）”，音频重编码策略可继续完善
- 屏幕推流（DXGI 预览链路与 `gdigrab` 直采链路）均支持音频双路采集与混音（每路优先 dshow，失败回退 native WASAPI）
- 摄像头推流（预览链路与直采链路）均支持音频双路采集与混音（每路优先 dshow，失败回退 native WASAPI）
- 除文件推流外，所有推流场景均为音视频分线程获取；文件模式继续沿用源文件内建音视频，不进入设备采集混音链路
- `gdigrab` 路径下受系统桌面合成和显卡环境影响，极端场景仍可能波动
- HDR/高级色彩场景下，DXGI 可能间歇返回 `R16G16B16A16_FLOAT` 帧；当前策略为跳过不支持帧并提示可切换 FFmpeg 后端，以换取稳定性
- 硬件编码依赖 FFmpeg 构建与驱动环境；`h264_nvenc/h264_amf` 不可用时会自动回退（auto）或按强制选项返回失败
- 纯 **copy** 模式下，部分 **H.264/HEVC 在 MP4 与 FLV 之间** 可能需要 bitstream filter；若遇失败，需按具体码流增加 bitstream 处理或改为重编码
- 地址历史仅会话内保留，未做持久化
- 日志是“最近片段”，不是完整历史

## 7. 本次问题复盘（2026-04）

### 8.1 现象

- 屏幕采集在 FFmpeg 后端可用，DXGI 后端出现黑屏/扭曲/推流失败（`-22`）
- 日志出现尺寸与 UI 枚举不一致（例如 `2561x1601` vs 实际帧 `2560x1600`）

### 8.2 根因

- 高 DPI 下 Qt 几何推导存在 1px 偏差，导致枚举尺寸与实际采集不一致
- HDR 场景下 DXGI 帧格式在 `B8G8R8A8_UNORM` 与 `R16G16B16A16_FLOAT` 间歇交替
- 旧逻辑把不支持格式当作故障触发重建设备，造成画面抖动/扭曲

### 8.3 修复策略

- DXGI 屏幕枚举优先使用 `DXGI OutputDesc` 原生坐标/分辨率
- 遇到不支持 HDR 浮点格式时跳帧而非重建
- 新增 HDR 弹窗提示，允许用户切换 FFmpeg 屏幕后端
- 推流链路自动跟随后端切换，无需手动改推流配置
- DXGI 日志限频，避免高频刷屏
