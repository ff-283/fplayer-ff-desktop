# 推拉流开发文档

## 1. 目标与范围（2026-04 更新）

本文档描述当前“推拉流”功能的工程落地方式，包括：

- 分层架构
- 关键接口
- UI 交互流程
- FFmpeg 库侧实现策略
- 已知限制与后续扩展建议

当前版本聚焦“可用闭环 + 可调优 + 分层可维护”，在进程内通过 **FFmpeg 动态库（libavformat / libavcodec / libavutil / libswscale / libswresample / libavdevice）** 实现推拉流，**不依赖**本机 `ffmpeg` 可执行文件或命令行。

实现类：`fplayer::StreamFFmpeg`

实现策略（当前）：

- 文件直推默认使用 **copy/remux**（低开销）
- 当配置了码率/尺寸/帧率时，文件模式切换为 **转码推流**（解码 -> 缩放 -> 编码 -> 推流）
- 屏幕模式支持两条视频链路：
  - `__screen_preview__`：消费 DXGI 预览帧总线（推荐，降低采集开销与鼠标闪烁）
  - `__screen_capture__`：FFmpeg `gdigrab` 采集链路（回退）
- 屏幕推流链路与当前屏幕采集后端联动：
  - 后端为 `Dxgi` 时优先走 `__screen_preview__`
  - 后端为 `FFmpeg` 时走 `__screen_capture__`
- 摄像头模式默认走“当前预览帧推流”，避免设备二次占用冲突
- 在 **独立工作线程** 中执行循环，避免阻塞 UI；`stop()` 通过 `AVFormatContext::interrupt_callback` 与原子标志请求中断
- 启动时调用 `avformat_network_init()`（仅一次）；输入侧在 `avformat_open_input` 前预分配 `AVFormatContext` 以便尽早挂上中断回调
- `recentLog` 为应用侧追加的简短状态/错误说明（带长度上限，避免无限增长）；`lastExitCode` 为 libav 错误码（成功为 `0`）

封装与编码策略：

- **推流**：输出封装为 FLV（RTMP/多数直播链路友好）
- **拉流**：由输出路径/协议自动推断容器
- **编码器**：优先 H.264，不可用时回退 MPEG4
- **视频编码器选择**：
  - UI 参数 `videoEncoder` 支持 `auto / cpu / nvenc / amf`
  - `auto` 策略：`h264_nvenc -> h264_amf -> libx264/H264/MPEG4`
  - 日志会输出实际命中编码器及硬件/软件属性
- **实时参数**：支持 `fps / size / bitrate`，并提供“跟随当前”策略
- **音频来源参数**：推流侧仅使用 `audio_in / audio_out`（输入/输出设备分别指定）
- **屏幕音频编码**：`gdigrab` 链路下音频走 `decode + swresample + AAC encode + mux`，避免 FLV 对输入音频格式不兼容导致 `-22`
- **非文件推流音视频解耦**：`__screen_preview__ / __screen_capture__ / __camera_preview__ / __camera_capture__ / __compose_scene__` 均采用“视频线程 + 音频线程”并行采集/编码，复用统一时间基写入复用器
- **双设备音频混音**：非文件推流链路可同时采集输入设备（麦克风）与输出设备（系统回采），经双 FIFO 对齐后混成单路 AAC 推出；次路不足时不阻塞主路，主路可补静音连续输出
- **音频诊断日志**：屏幕推流可输出主/副路读帧数、样本数、FIFO 深度与采集模式（dshow/wasapi），用于快速定位“输出设备无声”
- **时间戳策略**：屏幕/转码路径优先使用源帧时间戳并做单调修正，减轻播放端“忽快忽慢”
- **预览暂停与推流（屏幕/摄像头预览链路）**：`StreamFFmpeg::pushScreenPreviewLoop` / `pushCameraPreviewLoop` 若仅依赖 `ScreenFrameBus::snapshotIfNew` / 摄像头 `serial` 递增，则在 **预览暂停** 后总线不再更新，会导致长时间无视频包、拉流端一直缓冲。当前实现为：无新 `serial` 时回退 `snapshot()` 取 **最后一帧**，并按目标帧率重复编码（静帧），时间戳仍单调递增。
- **结构拆解（2026-04）**：
  - `streamffmpeg_helpers.*` 承载参数解析、推流场景路由、编码器候选选择与通用诊断逻辑
  - `audio_pipeline.*` 承载音频平面混音、第二路 FIFO 混音，以及“编码帧填充（含尾包补静音）”可复用处理，降低多场景重复代码
  - `StreamFFmpeg::startPush/startPull` 仅保留编排，线程启动细节下沉为私有函数

### 组合场景推流（`__compose_scene__:`）

- **入口**：推流输入 URL 以 `__compose_scene__:` 为前缀，其后为分号分隔的键值对场景描述（画布宽高、`srcN=...` 各层素材类型、屏幕 `sourceId`、归一化矩形等），由 UI（`CaptureWindow`）组装。
- **合成位置**：在 `StreamFFmpeg::pushComposeSceneLoop` 工作线程内，按场景层级从 `CameraFrameBus` / `ScreenFrameBus` 取 YUV420P 帧，缩放后画入编码用 `AVFrame`（近邻采样），**不是**截取预览窗口像素。
- **屏幕与文件多路**：`ScreenFrameBus` 按 `sourceId` 分通道；DXGI 通过 `Service::screenSetFrameBusSourceId` 发布；组合模式下的 **文件播放** 通过 `Service::setPlayerComposeStreamBusId` 由 `PlayerFFmpeg` 在 `queuePreviewYuv` 中同步发布到同一总线，推流端对 `screen` / `file` 图层均用 `snapshot(sourceId)` + 槽位 **contain** 合成，避免拉伸。
- **像素格式**：组合链路只向 **YUV420P 三平面 CPU 缓冲** 写入，编码器上下文固定 `AV_PIX_FMT_YUV420P`。硬编候选若优先 `NV12` 会与三平面写入/清屏逻辑不兼容（`data[2]` 为空曾导致崩溃），故组合推流仅选用声明支持 `yuv420p` 的编码器并强制该像素格式。
- **宽高比**：当 UI 画布（`scene_w/h`）与推流输出分辨率（`outsize`）**宽高比不一致** 时，合成阶段按 **contain** 将画布等比缩放到编码分辨率内并 **水平/垂直居中**，余下区域为黑边，避免对 X/Y 独立缩放导致拉流端画面与本地预览比例不一致。输出流显式写入 `sample_aspect_ratio = 1:1`（方形像素）。
- **单路素材**：每个摄像头/屏幕素材在其 **槽位矩形**（由预览中子窗口几何映射而来）内同样按 **contain** 绘制，与 `FGLWidget` 预览的「等比适配」一致，避免把 16:9 视频硬拉成槽位宽高比。
- **线程与缓冲区**：采集线程向 UI / 总线投递 `QByteArray` 时须 **深拷贝** 再跨线程排队（避免 Qt 隐式共享与采集侧复用缓冲并发写）；`ScreenFrameBus::publish` 对 Y/U/V 做拷贝后再入通道，避免推流线程与采集线程竞态。
- **多路摄像头隔离**：`CameraFrameBus` 已支持 `sourceId` 分通道；组合模式每路摄像头分配独立 `sourceId`，推流按 `sourceId` 读取，避免多摄像头素材抢同一帧总线导致闪烁。
- **日志**：组合推流与 DXGI/摄像头侧的排障日志默认关闭（`#if 0` 或注释），需要联调时再打开对应片段。
- **编码器私有参数兼容（2026-04）**：组合推流编码器初始化改为按编码器名分支设置参数，避免把 x264 参数误传给硬编导致启动失败：
  - `h264_nvenc`：`preset=p1, tune=ll, rc=cbr, zerolatency=1`
  - `h264_amf`：不写入 x264 风格 `preset/tune` 字符串（避免 `Unable to parse "preset/tune"`）
  - `libx264/H264(software)`：保留 `superfast/ultrafast + zerolatency` 低延迟配置

说明：文件模式默认不重编码；其它模式按配置与场景走编码链路。

### 组合模式 UI 与多素材调度（`widget/src/capturewindow.cpp`）

- **追加文件素材**：须先将 `ComposeSourceWidget` 放入 `QMdiArea` 子窗口并 `show()`（必要时 `QApplication::processEvents`），再 `Service::bindPlayerPreview` / `openMediaFile`，保证 `FVideoView`/`FGLWidget` 具有非零尺寸后再绑定预览（避免 OpenGL 预览黑屏）。
- **多路屏幕采集**：组合模式下每路屏幕素材按自身状态独立应用 `selectScreen / fps / cursor / active`，不再全局强制单路激活；`preferScreenRow/excludeScreenRow` 仅用于显式恢复/暂停目标行。
- **播放/暂停**：组合模式下 **不再** 全局暂停所有素材；底部工具栏播放/暂停在组合模式下 **只作用于当前选中的素材**，`toggleComposeSourcePlayPauseAt(m_composeSelectedIndex)` 与 `updateComposePlaybackIcons()` 随列表/画布选中切换同步 **该素材** 的播放状态与图标（不在每个子窗口上再放独立按钮）。
- **离开组合模式**：`suspendComposeSourcesForBackground()` 对全部素材执行 `playerPause` / `cameraPause` / `screenSetActive(false)`，并隐藏各 MDI 子窗口，保留 `m_composeSources` 以便再次进入时恢复布局。
- **素材标题同步**：组合模式下切换摄像头/屏幕设备时，同步更新子窗口标题与左侧列表（列表附短 `sourceId` 便于区分同类素材）。
- **直接进入组合模式追加屏幕素材（2026-04）**：`addComposeScreenSource()` 已补齐与“屏幕模式 selectScreen”一致的防护链路（HDR 检测提示、DXGI 失败自动回退 FFmpeg）。避免“未先进入屏幕模式时，首次追加屏幕素材卡住/未响应”。

联调配置示例（非文件推流）：

- 仅麦克风输入：`audio_in=麦克风阵列 (Realtek...)`，`audio_out=` 置空
- 仅系统声音：`audio_out=扬声器 (Realtek...)`，`audio_in=` 置空
- 双设备混音：`audio_in=麦克风阵列 (Realtek...)` + `audio_out=扬声器 (Realtek...)`

降级策略：

- 双路模式下单路失败 -> 自动降级单路继续推流
- 双路都失败 -> 自动降级无声推流
- 文件模式不使用 `audio_in/audio_out`，沿用源内建音视频

## 3. 关键流程

## 3.1 推流流程

1. 用户打开 `推流` 对话框
2. 配置输出地址与推流参数（fps/尺寸/码率/视频编码器/宽高比/输入设备/输出设备）
3. 调用 `Service::streamStartPushByScene(...)`
4. Service 根据场景组装推流请求并转发到 `IStream::startPush(...)`
5. `StreamFFmpeg` 在后台线程按策略执行（copy / transcode / capture / preview-frame）
6. UI 定时拉取状态与日志展示（包含“最终生效参数”）

## 3.2 拉流流程（监视窗口 + 独立预览）

1. 用户打开 `拉流` 监视窗口（非模态）
2. 配置协议模板、推流码、模式（监看/录制），录制模式下配置保存路径
3. 调用 `Service::streamStartPull(...)`
4. `StreamFFmpeg` 在后台线程中执行拉流转封装；监看模式下输出走 `null`，录制模式写目标文件
5. 同一拉流线程内：
   - 视频：解码并发布到 `ScreenFrameBus("pull_preview")`
   - 音频：解码 + 重采样 + 本地 `QAudioSink` 输出（构建可用时）
6. 预览窗口自动弹出并绑定 `pull_preview` 总线显示（不再播放器二次 `openMediaFile` 拉流）
7. 预览窗口暂停/音量通过 `IStream` 预览控制接口作用于拉流线程内部预览链路

## 3.3 退出拦截

主窗口关闭时，若存在运行中的推/拉流任务，UI 会提示：`当前正在执行推拉流任务，是否确定停止并退出`。确认后先 `streamStop()`，再关闭相关监视/预览窗口并退出程序。

## 5. 设计取舍

## 5.1 为何在进程内使用 libav* 而非调用 ffmpeg 进程

- 与项目已集成的 FFmpeg 动态库 **统一依赖**，安装包无需再携带或查找 `ffmpeg.exe`
- 行为在应用内可控，便于与 **中断、错误码、后续扩展（编码参数、多任务）** 结合
- 仍可通过 `lastError` / `lastExitCode` 对照 libav 错误排查问题

## 5.2 为何改为“拉流线程内解码 + 总线预览”

- 避免“拉流线程 + 预览播放器”对同一网络源二次连接导致的不稳定
- 降低停止/关闭窗口时的线程重入风险
- 预览控制（暂停/音量）可直接作用于同一条拉流链路

## 6. 已知限制

- 推流输出侧当前固定以 FLV 创建输出上下文；若需 RTSP/SRT 专用封装策略仍需继续细化
- 文件转码推流当前为“视频转码 + 音频流 copy（若源存在音频）”，音频重编码策略可继续完善
- 屏幕推流（DXGI 预览链路与 `gdigrab` 直采链路）均支持音频双路采集与混音（每路优先 dshow，失败回退 native WASAPI）
- 摄像头推流（预览链路与直采链路）均支持音频双路采集与混音（每路优先 dshow，失败回退 native WASAPI）
- 组合推流支持音频线程链路、双路采集与混音，并采用固定音频节拍 + 补静音策略降低抖动
- 除文件推流外，所有推流场景均为音视频分线程获取；文件模式继续沿用源文件内建音视频，不进入设备采集混音链路
- **文件直推 / 文件转码推流** 打开的媒体与 **预览播放器** 为独立管线：预览暂停、进度 **不会**自动同步到该推流输入；若需与预览画面帧级一致，需另行设计（例如组合模式经 `ScreenFrameBus` 等发布预览帧再推流）。
- `gdigrab` 路径下受系统桌面合成和显卡环境影响，极端场景仍可能波动
- HDR/高级色彩场景下，DXGI 可能间歇返回 `R16G16B16A16_FLOAT` 帧；当前策略为跳过不支持帧并提示可切换 FFmpeg 后端，以换取稳定性
- 硬件编码依赖 FFmpeg 构建与驱动环境；`h264_nvenc/h264_amf` 不可用时会自动回退（auto）或按强制选项返回失败
- 纯 **copy** 模式下，部分 **H.264/HEVC 在 MP4 与 FLV 之间** 可能需要 bitstream filter；若遇失败，需按具体码流增加 bitstream 处理或改为重编码
- 地址历史仅会话内保留，未做持久化
- 日志是“最近片段”，不是完整历史
- 预览音频当前依赖 `stream_ffmpeg` 目标可用音频输出依赖（QtMultimedia）；若构建环境缺失则自动降级为仅视频预览并写日志提示

## 6.1 近期修复记录（2026-04）

- **组合模式追加屏幕素材卡住**：根因是组合入口未复用屏幕模式的 HDR/DXGI 回退流程；现已统一，直接进入组合模式也会触发同等后端保护。
- **组合推流硬编参数不兼容**：根因是将 x264 `superfast/zerolatency` 参数传给 `h264_nvenc/h264_amf`；现已改为按编码器名分支设置，消除 `Unable to parse preset/tune`。
- **Release 默认暂停（Debug 正常）**：根因是 `CameraQt6::selectCamera()` 启动后未同步 `m_isPlaying=true`，`selectCameraFormat()` 可能 stop 后不再 start；现已修复并补充 `pause/resume` 空指针保护。

## 7. 本次问题复盘（2026-04）

### 8.1 现象

- 屏幕采集在 FFmpeg 后端可用，DXGI 后端出现黑屏/扭曲/推流失败（`-22`）
- 日志出现尺寸与 UI 枚举不一致（例如 `2561x1601` vs 实际帧 `2560x1600`）

### 8.2 根因

- 高 DPI 下 Qt 几何推导存在 1px 偏差，导致枚举尺寸与实际采集不一致
- HDR 场景下 DXGI 帧格式在 `B8G8R8A8_UNORM` 与 `R16G16B16A16_FLOAT` 间歇交替
- 旧逻辑把不支持格式当作故障触发重建设备，造成画面抖动/扭曲

### 8.3 修复策略

- DXGI 屏幕枚举优先使用 `DXGI OutputDesc` 原生坐标/分辨率
- 遇到不支持 HDR 浮点格式时跳帧而非重建
- 新增 HDR 弹窗提示，允许用户切换 FFmpeg 屏幕后端
- 推流链路自动跟随后端切换，无需手动改推流配置
- DXGI 日志限频，避免高频刷屏
